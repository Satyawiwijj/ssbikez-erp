## Financial / data-integrity

- [ ] **Vehicle List.price_lists** ‚Äî Missing: reference makes Price Lists mandatory on the vehicle stock record so a sale must be validated against an approved rate; Django's `CustomerPrice` master (`masters/models.py:311-321`) exists but is never referenced anywhere under `sales/` (confirmed by grep) ‚Äî `VehicleSalesOrderForm.total_amount`/`booking_amount`/`discount_amount` (`sales/forms.py:186,205-207`) are free-typed decimal fields with only a help-text hint, no cross-check against a price list. This is a control gap on every single New Vehicle Sales Order, not an edge case ‚Äî highest blast radius of the three items in this section. ‚Äî Open
- [ ] **New Vehicle Sales Order.customer / gst_category** ‚Äî Deviation: `VehicleSalesOrder.save()` (`sales/models.py:466-468`) only copies `customer.gst_category` onto the order `if not self.gst_category` ‚Äî i.e. once set, it is never refreshed. `gst_category` is also a plain editable field on the form (`sales/forms.py:186`), so it can be hand-overridden and then never re-synced. If a user changes the linked customer on an existing order, the stale GST category silently persists, which drives the CGST+SGST vs IGST split ‚Äî direct risk of an incorrectly taxed invoice with no error or warning. ‚Äî Open
- [ ] **Vehicle List.vehicle_code (New Vehicle Sales Order.vehicle)** ‚Äî Deviation: reference's Vehicle Code is a mandatory Link; Django's `VehicleSalesOrder.vehicle` FK is `null=True, blank=True` (`sales/models.py:363-368`), so a vehicle-sale order can be saved with no physical stock unit attached. `VehicleSalesOrderForm` help text (`sales/forms.py:178`) frames the blank case as intentional for spares/accessories-only orders, which narrows the real risk, but nothing distinguishes "spares order" from "vehicle order missing its stock link by mistake" at the model level ‚Äî a mis-saved vehicle order can leave the matching `VehicleStock` row un-reserved/un-sold, undetected. ‚Äî Open
- [x] **Invoice.cancel() doesn't reverse its auto-posted GL entry** ‚Äî Deviation found and fixed during Task 5, not from the Tier 1 matrix walkthrough (not a `PARITY_MATRIX.csv` row): `Invoice.submit()` (`billing/models.py`) auto-posts a `JournalEntry` (Dr Accounts Receivable / Cr Sales Revenue) via `post_journal_entry`, but `Invoice.cancel()` previously only flipped `docstatus` (inherited from `DocStatusMixin.cancel`) and never reversed that entry ‚Äî a cancelled invoice permanently overstated both accounts in the General Ledger. Also documented as an accepted risk in `CLIENT_GUIDE.md` ¬ß9/¬ß6.7 prior to this fix. ‚Äî Fixed: `Invoice.cancel()` now calls a new `reverse_journal_entry()` that posts a mirror-image `JournalEntry` (swapped debit/credit per line) tagged against the same invoice, guarded against double-reversal. See `billing/tests.py::InvoiceCancelReversesJournalEntryTests`.
- [x] **Payment reconciled to Completed never posted its GL entry** ‚Äî Deviation found and fixed during Task 7 (Tier 2, Billing/Payments/GL), not from a `PARITY_MATRIX.csv` row: `Payment.save()` (`billing/models.py:269-275`, pre-fix) only auto-posted the Dr Cash-or-Bank / Cr Accounts Receivable `JournalEntry` `if is_new and payment_status == COMPLETED` ‚Äî i.e. only for a Payment *created* already-Completed. `payment_reconciliation` (`billing/_gap14_31_views.py`, pre-fix) marked selected Pending payments Completed via a bulk `QuerySet.update()`, which never calls `.save()` at all. Net effect: any payment recorded Pending and later reconciled to Completed ‚Äî the normal cash-collection workflow ‚Äî showed as "Completed" in the UI and the Daily Collection Report with **no corresponding General Ledger entry ever posted**, silently understating both Cash/Bank and Accounts Receivable. The gap was already flagged in a code comment ("a known gap, noted in the report rather than silently patched over") but never fixed. Same class of bug as the Invoice.cancel() GL-reversal gap above. ‚Äî Fixed: `Payment.save()` now posts whenever `payment_status == COMPLETED` (dropped the `is_new` condition; `post_journal_entry()`'s own `hasattr(self, 'journal_entry')` guard keeps repeated saves idempotent), and `payment_reconciliation` now saves each reconciled `Payment` instance individually instead of a bulk `.update()`, so the model hook fires. See `billing/tests.py::PaymentReconciliationPostsJournalEntryTests`.
- [ ] **Customer Payment.customer_items ‚Äî no multi-invoice allocation** ‚Äî Missing: reference's Customer Payment lets one receipt allocate across several outstanding invoices/documents via a `customer_items` child table (`ref_docname`/`allocate_amount`/`balance_amount` per line, `reference_erp_spec/14_Receipts.md:20,199-219`). Django's `billing.Payment` (`billing/models.py:241-245`) is a single mandatory `ForeignKey` to exactly one `Invoice` ‚Äî there is no way to record one payment against multiple invoices, or a partial allocation split across documents. A cashier collecting one lump sum against several open invoices for the same customer has no single-record way to capture that in the current schema. ‚Äî Open
- [ ] **No Chart-of-Accounts / Bank / Mode-of-Payment master data** ‚Äî Missing: reference resolves `account_from`/`account_to` (Supplier Payment, Customer Payment, Ledger Creation) and `bank_name`/`to_account`/`from_account` as `Link` fields against an `Account` doctype (Chart of Accounts) and `pay_mode` against a `Mode of Payment` master. Confirmed by grep (`class Account\b|class Bank\b|Mode of Payment|ModeOfPayment` across all `.py` outside `reference_erp_spec/`) ‚Äî zero hits. `billing.JournalEntryLine.account` (`billing/models.py:522`) is a free-text `CharField` with no master-data validation, and `billing.Payment.payment_method` (`billing/models.py:246-250`) is a fixed `TextChoices` list, not a configurable master. Every GL account name in the system is typed by hand ‚Äî a stray "Accounts Receivable" vs "Accounts receivable" typo would silently split a ledger balance across two rows with no validation catching it. ‚Äî Open
- [ ] **No accounts-payable / Supplier Payment tracking** ‚Äî Missing: reference's `Supplier Payment` doctype (`reference_erp_spec/02_Payments.md`) tracks payments made *to* suppliers against outstanding purchase documents (`Supplier Pay Child` allocation table, cash/bank split, account resolution). Confirmed by grep ‚Äî zero hits for `Supplier Payment`/`SupplierPayment` anywhere under the Django app outside `reference_erp_spec/`. `masters.Supplier` (`masters/models.py:20`) is a master-data record only; `spares.PurchaseInvoice` (`spares/models.py:278`) records what's owed to a supplier but nothing in the codebase records or tracks what's actually been *paid* to a supplier against those invoices ‚Äî there is no accounts-payable ledger at all on the Django side. ‚Äî Open
- [x] **Counter Sale Return.godown ‚Äî return could be submitted without a warehouse, silently losing the returned stock (Tier 4)** ‚Äî Deviation found and fixed prior to this Tier 4 pass (commit `baa14d2`), now recorded in the matrix: `CounterSaleReturn.godown` (`spares/models.py:588`) is `on_delete=models.SET_NULL, null=True, blank=True`, and the create view always auto-submits a return immediately after saving. The stock-reversal `post_save` signal (`spares/models.py:664-698`, `on_counter_sale_return_status_changed`) silently no-ops when `godown_id` is `None` (`if not instance.godown_id: return`), so a return created without a godown was marked `submitted` while the returned stock was never credited back to any warehouse ledger ‚Äî lost entirely with no error surfaced. Reference makes `godown` mandatory YES on this doctype (`reference_erp_spec/13_Spares_Purchase.md:2274`). ‚Äî Fixed: `CounterSaleReturn.submit()` now raises `ValueError` without a godown, and `CounterSaleReturnForm.fields['godown'].required = True` catches it earlier as a normal form error. See `spares/tests.py::CounterSaleReturnGodownRequiredTests`.
- [ ] **Spares GST fields hardcoded at 9%/9%, not driven by CompanySettings or IGST-aware (Tier 4)** ‚Äî Deviation: every GST-bearing line model under `spares/` (`PurchaseInvoiceItem.sgst`/`.cgst` at `spares/models.py:412-413`, `TaxChargeLine.sgst`/`.cgst` at `:37-38`, `SparesIssueAlteration` lines at `:804-805`, `ServiceSparesIssueReturnItem` at `:1187-1188`/`:1261-1262`, `SparesIssueAlterationItem`/warranty-claim lines at `:1482-1483`) defaults `sgst`/`cgst` to a flat 9% each and computes tax as a fixed 50/50 split with no `igst` field anywhere in the app ‚Äî confirmed by grep, `spares/models.py` never imports `billing.models.split_gst`. This is the identical class of bug already found and fixed in `rto/models.py` (`NumberReceiptEntryCreation`, commit `fdb3740`): tax amounts are independent of `CompanySettings.cgst_rate`/`.sgst_rate` and there is no interstate/IGST handling for a spares sale to an out-of-state customer. Lower severity than the RTO instance was, because the reference `Spares Receipt Child`/`Counter Sale Return Child` doctypes themselves also have no `igst` field (confirmed by grep of `reference_erp_spec/13_Spares_Purchase.md`) ‚Äî so this is a Django-internal consistency gap against the now-centralized `split_gst()` helper used by billing/service/RTO, not a reference-parity gap. Recorded per Tier 4 scope, not fixed this pass (replicating the RTO fix across every GST-bearing spares line model is its own dedicated change). ‚Äî Open

## Workflow blockers

- [ ] **New Vehicle Sales Order.sales_enquiry_form + Branch.allow_without_enquiry_form** ‚Äî Missing: the reference app gates whether the enquiry-form link is mandatory per branch, live-verified as required at Coimbatore/Sulur-E/Tiruppur and not required at Gobi/Nambiyur/Puliampatti/Sathyamangalam/Sulur/Trichy. Django has no `allow_without_enquiry_form` field on `accounts.Branch` (`accounts/models.py:93-106`) and `VehicleSalesOrder.enquiry` is unconditionally optional for every branch (`sales/models.py:352-357`), with no branch-conditional requiredness logic in `sales/forms.py`. The branch-specific enquiry-tracking step of the sales workflow is currently unenforceable anywhere in the app. ‚Äî Open
- [ ] **Ledger Creation.approve + amended_from ‚Äî no approval gate or cancel/amend lifecycle on manual Journal Entries** ‚Äî Deviation/Missing (Tier 2): reference's `Ledger Creation` doctype is Submittable and its `before_submit` client script (`reference_erp_spec/14_Receipts.md:969-972`) throws unless the `approve` checkbox is set and both `from_account`/`to_account` are filled ‚Äî a manual voucher cannot go live without an approval step, and as a Submittable doctype it can later be cancelled/amended. `billing.JournalEntry` (`billing/models.py:464`) is a plain `models.Model` with **no docstatus field at all** ‚Äî confirmed: only `billing.Invoice` extends `DocStatusMixin` (`billing/models.py:6,45`). `journal_entry_create` (`billing/_gap14_31_views.py:166-192`) only checks that debit totals equal credit totals; there is no approval gate, and `billing/urls.py:60-62` wires only list/create/detail for JournalEntry ‚Äî no update or cancel route. Once a manual journal entry is saved, a mis-posted GL line can only be corrected by posting a second, manually-balanced counter-entry ‚Äî there is no reversal mechanism the way `Invoice.cancel()` now has via `reverse_journal_entry()`. ‚Äî Open
- [ ] **Ledger Creation Date Master.allowed_days ‚Äî no backdating window on manual GL postings** ‚Äî Missing: reference's singleton `Ledger Creation Date Master` doctype restricts how many days back a `Ledger Creation` entry's date can be set, enforced client-side in `before_save` (`reference_erp_spec/14_Receipts.md:948-967`). No equivalent setting exists on `accounts.CompanySettings` (`accounts/models.py:163-215`) or `billing.forms.JournalEntryForm` (`billing/forms.py:278-296`, no `clean_entry_date`) ‚Äî a manual Journal Entry's `entry_date` accepts any date with no min/max validation, so a user can freely backdate GL postings into an already-closed period. ‚Äî Open
- [ ] **Job Card service_status pipeline ‚Äî no stage-order enforcement (Tier 3)** ‚Äî Deviation: `PRODUCTION_DELIVERY_REPORT.md:151` documents the Job Card pipeline as `Pending -> Water Wash -> In Bay -> In Progress -> Outwork (if needed) -> Final Inspection -> Ready -> Invoiced`, and `service/models.py` does correctly reimplement the reference's chain-of-separate-Submittable-doctypes design as one `service_status` field advanced by `post_save` signals on each stage document (`WaterWashDone`/`BayInCreation`/`BayOutCreation`/`OutworkEntryIssue`/`OutworkEntryReturn`/`FinalInspection`, signals at `service/models.py:1139-1173`). But nothing enforces the *order* those stage documents get submitted in: `_advance_job_card_status` (`service/models.py:767-770`) unconditionally overwrites `service_status` whenever it differs from the incoming value, and the stage `_create` views (e.g. `bay_out_create`, `service/views.py:1070-1080`) have no precondition checking the Job Card's current `service_status` before allowing a new stage document. A floor supervisor can submit a Bay Out Creation before any Water Wash or Bay In document exists, jumping the Job Card straight from Pending to In Progress, or regress it backward to an earlier stage if an earlier-stage document is (re)submitted after a later one already advanced it. ‚Äî Open
- [ ] **jobcard_status_update ‚Äî manual status override bypasses the entire stage-document pipeline (Tier 3)** ‚Äî Deviation: `service/views.py:362-385` exposes a quick-action view (used from `jobcard_detail.html`) that lets any user with `service`/`edit` permission and ownership of the Job Card POST an arbitrary `service_status` value ‚Äî any pipeline stage, forward or backward ‚Äî with no stage document (Water Wash Done, Bay In/Out Creation, etc.) ever created or submitted. The only guard is that status is locked once a Service Invoice exists (line 375). The reference workflow has no equivalent manual-override field at all ‚Äî status there is purely a byproduct of which stage doctype was last submitted, which is exactly the audit trail this escape hatch bypasses. ‚Äî Open
