## Role & Permission Management (Tier 8)

- [ ] **User.roles — single-role-per-user instead of the reference's multi-role assignment (Tier 8)** — Deviation: the reference `User` doctype's `roles` field is a `Table` of `Has Role` rows (`reference_erp_spec/30_standard_doctypes_full_schema.md:10170`), Frappe's standard many-to-many — any user can simultaneously hold several roles (e.g. both "Cashier" and "Accounts") and the effective permission set is the union of all of them. `accounts.User.role` (`accounts/models.py:132-137`) is a single nullable `ForeignKey` to `Role` — a Django user can hold exactly one role at a time. Every downstream permission check in this app (`accounts/permissions.py:36-44,74-81,122-144`) reads `user.role` singular, so this is load-bearing: a real job function that spans two of the reference's `ROLE_PERMISSIONS` buckets (e.g. someone who does both cashiering and accounts-ledger work) cannot be modeled without either creating a combined ad-hoc role that isn't in the reference's role vocabulary, or under-provisioning one side of their job. Directly related: reference's `role_profile_name` (Link to Role Profile, `...:10168`) — a preset bundle of roles assignable in one step — has no Django equivalent either, since there is nothing to bundle with only one role slot. — Open
- [ ] **ModulePermission — module-level granularity instead of the reference's per-DocType permission matrix, no permission level, no if_owner (Tier 8)** — Deviation: the reference's Role Permission Manager (`DocPerm`/`Custom DocPerm`, standard Frappe framework tables backing every one of the ~100+ business doctypes this app reimplements) grants read/write/create/delete/submit/cancel/amend/print/email/report/import/export/share **per individual DocType per Role**, with a numeric permission level (0-9, for field-level restriction) and an `if_owner` flag (restrict a permission to only records the user themself created). `accounts.ModulePermission` (`accounts/models.py:391,407-439`) grants only 4 actions (`can_view`/`can_create`/`can_edit`/`can_delete`) keyed by a coarse `module` bucket (`MODULE_CHOICES`, ~10 app-level groupings like `'sales'`/`'service'`/`'finance'` — see `accounts/permissions.py:114-119,122-144` for the enforcement, wired via `@require_module_action` across `billing/`, `customers/`, `masters/`, `rto/`, `sales/`, `service/`, `spares/`, `used_vehicles/`, `vas/`). There is no submit/cancel/amend distinction (every `DocStatusMixin` submit/cancel action in this app is gated only by the same `'edit'` or `'create'` flag as an ordinary field edit), no permission level, and no `if_owner` restriction on the permission row itself — record ownership is instead handled by a separate, differently-scoped helper (`user_owns()`, `accounts/permissions.py:84-101`) that isn't wired into `require_module_action` at all. Net effect: the reference can grant "Cashier may create and submit a Customer Payment but not cancel one, and only sees payments they personally took"; Django can only express "Cashier can/cannot edit anything in the `billing` module." — Open
- [ ] **ModulePermission.role — no per-user module override, reference's module_profile/block_modules has no Django equivalent (Tier 8)** — Missing: the reference `User` doctype has `module_profile` (Link to Module Profile, `reference_erp_spec/30_standard_doctypes_full_schema.md:10171`) and `block_modules` (Table of Block Module, `...:10173`) — both per-*user* overrides of which modules are visible, independent of the user's Role. `accounts.ModulePermission.role` (`accounts/models.py:419`) is keyed by `Role` only, with no per-user row at all — every user sharing a Role gets identical module access with no way to individually widen or narrow it (e.g. temporarily block one Sales Executive from `billing` without touching the whole "Sales Executive" role). — Open
- [ ] **Branch.abbreviation — missing (Tier 8)** — Missing: reference's `Branch` doctype has a mandatory `abbreviation` field alongside `branch` (`reference_erp_spec/30_standard_doctypes_full_schema.md:307-312`), used as a short code prefix in the reference's naming series for branch-scoped documents. `accounts.Branch` (`accounts/models.py:93-99`) has no `abbreviation` field at all (`branch_name`, `address`, `phone`, `gstin`, `is_active`, `created_at` only) — cosmetic on its own, but consistent with this app's broader pattern of auto-integer-PK document numbering rather than the reference's configurable prefixed naming series (see the existing `Customer/Supplier Payment naming_series` item below). — Open
