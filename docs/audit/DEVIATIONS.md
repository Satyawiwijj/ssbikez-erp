## Financial / data-integrity

- [ ] **Vehicle List.price_lists** — Missing: reference makes Price Lists mandatory on the vehicle stock record so a sale must be validated against an approved rate; Django's `CustomerPrice` master (`masters/models.py:311-321`) exists but is never referenced anywhere under `sales/` (confirmed by grep) — `VehicleSalesOrderForm.total_amount`/`booking_amount`/`discount_amount` (`sales/forms.py:186,205-207`) are free-typed decimal fields with only a help-text hint, no cross-check against a price list. This is a control gap on every single New Vehicle Sales Order, not an edge case — highest blast radius of the three items in this section. — Open
- [ ] **New Vehicle Sales Order.customer / gst_category** — Deviation: `VehicleSalesOrder.save()` (`sales/models.py:466-468`) only copies `customer.gst_category` onto the order `if not self.gst_category` — i.e. once set, it is never refreshed. `gst_category` is also a plain editable field on the form (`sales/forms.py:186`), so it can be hand-overridden and then never re-synced. If a user changes the linked customer on an existing order, the stale GST category silently persists, which drives the CGST+SGST vs IGST split — direct risk of an incorrectly taxed invoice with no error or warning. — Open
- [ ] **Vehicle List.vehicle_code (New Vehicle Sales Order.vehicle)** — Deviation: reference's Vehicle Code is a mandatory Link; Django's `VehicleSalesOrder.vehicle` FK is `null=True, blank=True` (`sales/models.py:363-368`), so a vehicle-sale order can be saved with no physical stock unit attached. `VehicleSalesOrderForm` help text (`sales/forms.py:178`) frames the blank case as intentional for spares/accessories-only orders, which narrows the real risk, but nothing distinguishes "spares order" from "vehicle order missing its stock link by mistake" at the model level — a mis-saved vehicle order can leave the matching `VehicleStock` row un-reserved/un-sold, undetected. — Open
- [x] **Invoice.cancel() doesn't reverse its auto-posted GL entry** — Deviation found and fixed during Task 5, not from the Tier 1 matrix walkthrough (not a `PARITY_MATRIX.csv` row): `Invoice.submit()` (`billing/models.py`) auto-posts a `JournalEntry` (Dr Accounts Receivable / Cr Sales Revenue) via `post_journal_entry`, but `Invoice.cancel()` previously only flipped `docstatus` (inherited from `DocStatusMixin.cancel`) and never reversed that entry — a cancelled invoice permanently overstated both accounts in the General Ledger. Also documented as an accepted risk in `CLIENT_GUIDE.md` §9/§6.7 prior to this fix. — Fixed: `Invoice.cancel()` now calls a new `reverse_journal_entry()` that posts a mirror-image `JournalEntry` (swapped debit/credit per line) tagged against the same invoice, guarded against double-reversal. See `billing/tests.py::InvoiceCancelReversesJournalEntryTests`.
- [x] **Payment reconciled to Completed never posted its GL entry** — Deviation found and fixed during Task 7 (Tier 2, Billing/Payments/GL), not from a `PARITY_MATRIX.csv` row: `Payment.save()` (`billing/models.py:269-275`, pre-fix) only auto-posted the Dr Cash-or-Bank / Cr Accounts Receivable `JournalEntry` `if is_new and payment_status == COMPLETED` — i.e. only for a Payment *created* already-Completed. `payment_reconciliation` (`billing/_gap14_31_views.py`, pre-fix) marked selected Pending payments Completed via a bulk `QuerySet.update()`, which never calls `.save()` at all. Net effect: any payment recorded Pending and later reconciled to Completed — the normal cash-collection workflow — showed as "Completed" in the UI and the Daily Collection Report with **no corresponding General Ledger entry ever posted**, silently understating both Cash/Bank and Accounts Receivable. The gap was already flagged in a code comment ("a known gap, noted in the report rather than silently patched over") but never fixed. Same class of bug as the Invoice.cancel() GL-reversal gap above. — Fixed: `Payment.save()` now posts whenever `payment_status == COMPLETED` (dropped the `is_new` condition; `post_journal_entry()`'s own `hasattr(self, 'journal_entry')` guard keeps repeated saves idempotent), and `payment_reconciliation` now saves each reconciled `Payment` instance individually instead of a bulk `.update()`, so the model hook fires. See `billing/tests.py::PaymentReconciliationPostsJournalEntryTests`.
- [ ] **Customer Payment.customer_items — no multi-invoice allocation** — Missing: reference's Customer Payment lets one receipt allocate across several outstanding invoices/documents via a `customer_items` child table (`ref_docname`/`allocate_amount`/`balance_amount` per line, `reference_erp_spec/14_Receipts.md:20,199-219`). Django's `billing.Payment` (`billing/models.py:241-245`) is a single mandatory `ForeignKey` to exactly one `Invoice` — there is no way to record one payment against multiple invoices, or a partial allocation split across documents. A cashier collecting one lump sum against several open invoices for the same customer has no single-record way to capture that in the current schema. — Open
- [ ] **No Chart-of-Accounts / Bank / Mode-of-Payment master data** — Missing: reference resolves `account_from`/`account_to` (Supplier Payment, Customer Payment, Ledger Creation) and `bank_name`/`to_account`/`from_account` as `Link` fields against an `Account` doctype (Chart of Accounts) and `pay_mode` against a `Mode of Payment` master. Confirmed by grep (`class Account\b|class Bank\b|Mode of Payment|ModeOfPayment` across all `.py` outside `reference_erp_spec/`) — zero hits. `billing.JournalEntryLine.account` (`billing/models.py:522`) is a free-text `CharField` with no master-data validation, and `billing.Payment.payment_method` (`billing/models.py:246-250`) is a fixed `TextChoices` list, not a configurable master. Every GL account name in the system is typed by hand — a stray "Accounts Receivable" vs "Accounts receivable" typo would silently split a ledger balance across two rows with no validation catching it. — Open
- [ ] **No accounts-payable / Supplier Payment tracking** — Missing: reference's `Supplier Payment` doctype (`reference_erp_spec/02_Payments.md`) tracks payments made *to* suppliers against outstanding purchase documents (`Supplier Pay Child` allocation table, cash/bank split, account resolution). Confirmed by grep — zero hits for `Supplier Payment`/`SupplierPayment` anywhere under the Django app outside `reference_erp_spec/`. `masters.Supplier` (`masters/models.py:20`) is a master-data record only; `spares.PurchaseInvoice` (`spares/models.py:278`) records what's owed to a supplier but nothing in the codebase records or tracks what's actually been *paid* to a supplier against those invoices — there is no accounts-payable ledger at all on the Django side. — Open

## Workflow blockers

- [ ] **New Vehicle Sales Order.sales_enquiry_form + Branch.allow_without_enquiry_form** — Missing: the reference app gates whether the enquiry-form link is mandatory per branch, live-verified as required at Coimbatore/Sulur-E/Tiruppur and not required at Gobi/Nambiyur/Puliampatti/Sathyamangalam/Sulur/Trichy. Django has no `allow_without_enquiry_form` field on `accounts.Branch` (`accounts/models.py:93-106`) and `VehicleSalesOrder.enquiry` is unconditionally optional for every branch (`sales/models.py:352-357`), with no branch-conditional requiredness logic in `sales/forms.py`. The branch-specific enquiry-tracking step of the sales workflow is currently unenforceable anywhere in the app. — Open
- [ ] **Ledger Creation.approve + amended_from — no approval gate or cancel/amend lifecycle on manual Journal Entries** — Deviation/Missing (Tier 2): reference's `Ledger Creation` doctype is Submittable and its `before_submit` client script (`reference_erp_spec/14_Receipts.md:969-972`) throws unless the `approve` checkbox is set and both `from_account`/`to_account` are filled — a manual voucher cannot go live without an approval step, and as a Submittable doctype it can later be cancelled/amended. `billing.JournalEntry` (`billing/models.py:464`) is a plain `models.Model` with **no docstatus field at all** — confirmed: only `billing.Invoice` extends `DocStatusMixin` (`billing/models.py:6,45`). `journal_entry_create` (`billing/_gap14_31_views.py:166-192`) only checks that debit totals equal credit totals; there is no approval gate, and `billing/urls.py:60-62` wires only list/create/detail for JournalEntry — no update or cancel route. Once a manual journal entry is saved, a mis-posted GL line can only be corrected by posting a second, manually-balanced counter-entry — there is no reversal mechanism the way `Invoice.cancel()` now has via `reverse_journal_entry()`. — Open
- [ ] **Ledger Creation Date Master.allowed_days — no backdating window on manual GL postings** — Missing: reference's singleton `Ledger Creation Date Master` doctype restricts how many days back a `Ledger Creation` entry's date can be set, enforced client-side in `before_save` (`reference_erp_spec/14_Receipts.md:948-967`). No equivalent setting exists on `accounts.CompanySettings` (`accounts/models.py:163-215`) or `billing.forms.JournalEntryForm` (`billing/forms.py:278-296`, no `clean_entry_date`) — a manual Journal Entry's `entry_date` accepts any date with no min/max validation, so a user can freely backdate GL postings into an already-closed period. — Open

## Cosmetic / low-impact

- [ ] **New Vehicle Sales Order.phone_number** — Missing: no `phone_number` field on `VehicleSalesOrder` (`sales/models.py:352-464`). The phone number is still available live via the linked `customers.Customer.phone` (`customers/models.py:23`); this only means the order doesn't keep its own point-in-time snapshot if the customer's phone is later edited. — Open
- [ ] **Vehicle List.vehicle_color** — Deviation: reference derives color from a Customer Price/price-list selection; Django models `color` as a direct field on the physical stock unit (`customers/models.py:100`). Functionally equivalent for recording the vehicle's color — this is a modeling-approach difference, not a missing capability. — Open
- [ ] **Vehicle List.sub_group** — Missing: no `sub_group` field on `customers.VehicleStock` (`customers/models.py:81-106`), though `masters.CustomerPrice.sub_group` exists for price-list grouping. Since `CustomerPrice` isn't wired into the sales flow at all yet (see the `price_lists` item above), this has no live business-logic impact today. — Open
- [ ] **Customer Payment.payment_type / pay_mode — mandatory in reference, optional in Django** — Deviation (Tier 2): reference makes both `payment_type` (Pay Type: Cash/Bank) and `pay_mode` mandatory, and uses them to gate which accounts get resolved before save (`reference_erp_spec/14_Receipts.md:337-370`). `billing.Payment.payment_method` (`billing/models.py:246-250`) is `blank=True, null=True` — a Payment can be recorded with no payment method at all. Lower impact than the reconciliation GL gap above because it doesn't silently lose GL postings, just weakens a data-capture guarantee. — Open
- [ ] **Customer/Supplier Payment naming_series** — Deviation: reference uses a configurable document naming series (`CUS-PAY-`/`SUP-PAY-`); Django displays `PAY-{pk}` from the auto integer primary key (`billing/models.py:267`). Cosmetic only. — Open
